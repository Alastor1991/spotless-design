# Spotless

**Музыка — людям. Деньги — артистам.**

Открытый музыкальный сервис: функционально — стриминг уровня Spotify Premium, идеологически — его противоположность. Пользователь сам решает, сколько платить в месяц, и эти деньги напрямую, без посредников и скрытых комиссий, распределяются между артистами, которых он реально слушал.

Этот репозиторий — **дизайн-прототип** (HTML/React, без бэкенда). Он является эталоном для будущей реализации: клиент — Rust + Slint, инфраструктура — Go, деньги — Solana (детали и обоснования в 5.5). Прототип должен быть доведён до состояния «доделывать максимум пару процентов».

> **Обновлено 12.06.2026.** Зафиксированы решения по реализации: **Slint** вместо iced/libcosmic, **Solana/USDC** вместо TON, **он-рамп-провайдеры** вместо PayPal-подписки, **Telegram Mini Apps отменены**. Прототипа это касается только текстами и парой экранов — конкретные правки внесены в чек-лист 3.3, контекст — в 5.5. Не воспроизводить старые формулировки («автосписание через PayPal», упоминания TON) в новых текстах.

## 1. Идеология

### 1.1. «Обратный столлманизм»
Проект под AGPL — но **не из идеологии, а чтобы вам было удобно**. Это центральный принцип всех текстов и всего UX:

- Пользователь **не обязан знать** про крипту, p2p, смарт-контракты, DHT и прочие «страшные слова». В основном интерфейсе их нет вообще.
- Никакого снобизма свободного ПО: ни «а чё вики не прочёл», ни терминов без объяснения, ни «ты должен был знать». Каждое объяснение — человеческим языком.
- Свобода продаётся как **удобство**: «нет скрытых комиссий», «деньги идут напрямую», «все функции доступны всем».
- При этом **технически подкованных не забыли**: в настройках есть хорошо видимый (не спрятанный!) раздел «Продвинутое» с сид-фразой, адресом кошелька, транзакциями, статусом сети, конфигом. Плюс пасхалки в теме «Студия» (см. 4.2). Таких «приветов адептам» должно быть несколько по всему продукту.

### 1.2. Честная монетизация
- Пользователь кладёт в месяц **сколько хочет**: пресеты $2 / $5 / $10 или своя сумма.
- Пополнение — **два равноправных пути**, поданных в том же стиле, что карточки входа (см. 1.4):
  1. **С карты** — виджет он-рамп-провайдера: Apple Pay / Google Pay / карта / SEPA (PayPal — просто один из методов внутри виджета, не основа). Подтверждение личности (KYC) — один раз, у провайдера, не у нас. Формулировка: «Один раз подтвердить личность — дальше в два касания».
  2. **С другого крипто-кошелька** — прямой перевод на свой адрес (QR / Solana Pay). «Подтверждение личности не понадобится». Обязательная жёсткая подпись «USDC в сети Solana» + пояснение в духе «Откуда эта цифра?» — перевод не в той сети это потеря денег.
- **Автосписания нет** — у он-рампов не существует подписочного API. Вместо него: ежемесячное напоминание + сохранённый метод оплаты у провайдера = пополнение в два касания. Тексты UI говорят «напоминание», а не «автосписание» (правка против текущего прототипа — см. 3.3).
- Сумма **ни на что не влияет**, кроме размера поддержки артистов. Все функции — всем и всегда. Это проговаривается явно в онбординге и в «Моей поддержке».
- Распределение — **пропорционально времени прослушивания, только между теми, кого слушал этот пользователь** (через смарт-контракты; в UI это слово не используется).
- Дополнительно: разовый донат по треку / альбому / синглу / артисту «просто так» — со счёта аккаунта, маленькой кнопкой `$` (дух «как на Бусти», ненавязчиво).
- «Без скрытых комиссий» — буквально: сетевые издержки показываются до подтверждения (на Solana это доли цента, текущая подпись «~$0.01» — честный потолок), а в фиатном пути виджет показывает **итог к зачислению** (landed cost) — комиссия провайдера (1–4%) не прячется.

### 1.3. Прозрачность артистов
- На странице артиста — **скромная строчка** в шапке: «Получает в среднем $X с одного слушателя · $Y за последний месяц» + ссылка «Откуда эта цифра?».
- Артист может скрыть месячную сумму со своей стороны (пример в прототипе: Velvet Antenna) — тогда показывается только среднее с слушателя.

### 1.4. Вход без барьеров
- **Google-аккаунт**: кошелёк создаётся автоматически, сид-фраза сохраняется в Google-аккаунт пользователя. «Ничего записывать не нужно».
- **Сид-фраза**: для тех, кто хранит ключи сам. «Гугл не понадобится».
- Оба пути равноправны и показаны на одном экране онбординга с честным объяснением каждого.
- Инвариант (юридический, не технический нюанс): сид-фраза при Google-входе лежит в Google-аккаунте **пользователя**; на инфраструктуру проекта ключи не попадают никогда и ни в каком виде (см. 5.5, «Право»).

### 1.5. Тон текстов
Сдержанно-приветливый, тёплый, без сюсюканья и без канцелярита. Фразы из онбординга — почти готовые слоганы будущей рекламной кампании: «Музыка — людям. Деньги — артистам», «Любая цифра — честная», «Дойдёт напрямую, целиком».

## 2. Название

**Spotless** — англ. «безупречный, без единого пятнышка» (= «без скрытых комиссий», чистая система). Второй смысл, который доходит на полтона позже: *Spot-less* — «без Спота». Добрый троллинг Spotify, без агрессии. Название утверждено.

## 3. Функциональный охват

Принцип: **ничего из функций Spotify не должно потеряться. Абсолютно.** Функционально интерфейс почти идентичен Spotify (привычка пользователя — святое), визуально — собственный язык (копировать фирменный дизайн Spotify нельзя).

### 3.1. Реализовано в прототипе
| Область | Что есть |
|---|---|
| **Онбординг** | 5 шагов: манифест → вход (Google / сид-фраза) → выбор темы из 6 → месячная сумма → 3 карточки-обучалки (сердечко, плюсик, $). Всё скипается; повтор — в настройках |
| **Главная** | Приветствие по времени суток, лента друзей (кто что слушает), Jam-кнопка, «Недавно слушали», «Новое от ваших артистов», «Собрано для вас», чипы Всё/Музыка/Подкасты |
| **Поиск** | Пустое состояние (история + жанры/настроения), живой поиск: лучший результат, треки, артисты, альбомы. ⌘K-бейдж в теме «Студия» |
| **Медиатека** | Фильтры Всё/Плейлисты/Артисты/Альбомы/Скачанное, закреплённые, сортировка, метки скачанного (живые, от реального состояния) |
| **Альбом/сингл/EP** | Шапка, play, перемешивание (3 режима), плюс, загрузка, «Поддержать релиз», шеринг, список треков с прослушиваниями, «Ещё от артиста» |
| **Артист** | Шапка со строчкой доходов, подписка, донат, популярные треки, «Об артисте», дискография, похожие |
| **Плеер-островок** | Капсула (или скруглённый — настройка), обложка→полный экран, лайк/плюс/$, prev/play/next, shuffle ×3, repeat, прогресс, текст, устройства, очередь, громкость, разворот |
| **Полный экран** | Градиент от обложки, крупная обложка, синхронный караоке-текст (клик по строке = перемотка), все управления |
| **Правая панель** | «Сейчас играет» (обложка, действия, далее в очереди, об артисте) / «Очередь» |
| **Оценка 1–10** | Long-press на сердечке → шкала. Клик — обычный лайк. Оценка ≥5: сердечко закрашено; <5: зачёркнуто, красноватое (антирейтинг). Бейдж с цифрой. Оценки «делают рекомендации точнее» |
| **Плюсик** | Поповер «В плейлист» со списком плейлистов, галочки, тосты добавления/удаления. Отдельная кнопка от лайка |
| **Донат `$`** | Модалка: пресеты $1/2/5/10 + своя сумма, баланс счёта, «Дойдёт напрямую, целиком», анимация благодарности. Вызов: строка трека, островок, страницы релиза/артиста, правая панель |
| **Моя поддержка** | Сумма/месяц (пресеты + своя капсула, фикс по клику вне поля), баланс + пополнение, история по месяцам с доп.донатами, полоса распределения по артистам + раскрытие (минуты, прослушивания, переходы), «без скрытых комиссий». ⚠ Тексты про PayPal-автосписание устарели — см. 3.3 |
| **Загрузки** | Кнопка-кружок: анимация заполнения контура → галочка (как в Spotify). Общее состояние: шапка релиза ↔ треки связаны двусторонне, каскадная загрузка альбома, отмена повторным кликом |
| **Настройки** | Внешний вид (6 тем, обои, акцент, плотность, форма островка, галерея тем, экспорт/импорт JSON), Воспроизведение и загрузки (качество потока/загрузок до FLAC 24/96, лимитное соединение, папка, кэш, нормализация, кроссфейд, gapless), Язык и регион (RU/EN, валюта по IP с запоминанием), Подсказки (повтор тура, подробные карточки), Аккаунт (публичный профиль, видимость доходов), **Продвинутое** (сид-фраза, адрес, транзакции, сеть, конфиг) |
| **Темы** | 6 на онбординге: Тёплая ночь / Студия / Глубина / Солнечная / Бумага / Зефир. Тема меняет цвета, шрифт, скругления (см. 4) |
| **i18n** | RU/EN целиком, названия треков всегда оригинальные |
| **Адаптив** | См. 5.3 — выдерживает сжатие до ~270px ширины плеера |
| **Персистентность** | Всё состояние в localStorage (`spotless_proto_v1`): тема, язык, оценки, лайки, плейлисты, загрузки, сумма, маршрут, сайдбар |

### 3.2. Решения по UX, которые уже приняты (не менять без обсуждения)
- Сердечко и плюсик — **раздельные** кнопки (в Spotify плюс совмещён с лайком — у нас нет).
- Кнопки в строке трека (десктоп): по наведению; у лайкнутых/оценённых сердечко видно всегда. Порядок: ♥ · ＋ · ⭳ · $.
- Умное перемешивание — третий режим кнопки shuffle со звёздочкой (дизайн-паттерн Spotify) + тост.
- «Медиатека» — именно это слово в RU (не «Библиотека»).
- Подкасты — вторичны: чип на главной + заглушка «скоро» (+ обещание поддержки авторов). Полноценно — потом.
- Социальное: лента друзей, совместные плейлисты (бейдж на плейлисте), публичные профили, шеринг ссылками, Jam — заложены в UI, экраны частично TODO.
- Валюта: $ по умолчанию, смена в настройках одной кнопкой, запоминается насовсем.
- Островок-плеер — плавающая капсула с отступами (не приклеенная полоса); форма переключается в настройках.
- Пополнение счёта — **два пути** (фиат-виджет / перевод с кошелька), поданные карточками в стиле экрана входа. Решение зафиксировано (см. 1.2), экран — в TODO.

## 4. Дизайн-система

### 4.1. Происхождение
Три направления (A «Тёплая бумага», B «Студия», C «Солнечная») → утверждён **гибрид**: структура и мягкость A + лёгкие разделители и пасхалки B + капсулы и плавающий островок C. Файл с направлениями: `Стилевые направления.html` (исторический артефакт, не трогать).

### 4.2. Темы — это не только цвета
Тема переключает **цвета + шрифт + скругления + характер**:

| Тема | id | Фон | Шрифт | Особенности |
|---|---|---|---|---|
| Тёплая ночь (дефолт) | `dark-warm` | тёплый графит | Golos Text | базовая тёмная |
| Студия | `dark-neutral` | холодный графит | Source Sans 3 + JetBrains Mono | **пасхалки**: `v0.4` у лого, `cache 1.2 GB · 12 peers · dht ok` в сайдбаре, ⌘K в поиске, бейдж `FLAC 24/96` в плеере, mono-цифры |
| Глубина | `dark-sea` | тёплый графит | Golos Text | голубой акцент |
| Солнечная | `light-cream` | кремовый | Nunito Sans | базовая светлая |
| Бумага | `light-neutral` | нейтральный белый | Nunito Sans | |
| Зефир | `light-rose` | розоватый | Nunito Sans | розовый акцент |

Акцент по умолчанию — **жёлтый** `oklch(0.78–0.80 0.13 95)`; в настройках подменяется (жёлтый/голубой/розовый/зелёный) поверх любой темы.

Все цвета — **только oklch** и только через CSS-переменные (`--bg0..3, --line, --text, --mut, --acc, --acc-ink, --island, --bad...`). Никаких хардкод-цветов в компонентах. (Эта дисциплина — не педантизм: переменные станут машинным мостом в Slint-темы, см. 6/P1.)

### 4.3. Кастомизация («злее Discord Nitro»)
Заложено: темы, обои (свои картинки/градиенты/паттерны), акцент, плотность, форма островка, шрифты, экспорт/импорт/шеринг тем, галерея тем сообщества, отдельная кастомизация плеера. Реализовано в прототипе частично (см. 3.3). Это **визитная карточка** продукта — доступна всем бесплатно.

### 4.4. Обложки
Реальных обложек нет — полосатые заглушки (`Cover`, повёрнутый repeating-linear-gradient, тон через `hue`, светлота от темы `--cover-l1/l2`, подпись mono «cover»/«album art»). Не заменять на SVG-арт; при выходе — реальные обложки.

### 4.5. Контент-вселенная (моки)
Артисты: Молоко и мёд (Казань, инди), Low Tide Choir (Брайтон, dream-pop), Velvet Antenna (synthwave, скрывает доходы), Paper Planes Club (Лиссабон, инди-рок), Танец Бруно (Москва, электроника), Грунтовка (Пермь, пост-панк). Пользователь — Лиза, $5/мес. Караоке-текст «Стеклянной воды» — оригинальный, написан для прототипа. Все названия/тексты — выдуманные, ничего лицензированного.

## 5. Техническая часть прототипа

### 5.1. Стек
- React 18.3.1 UMD + Babel standalone (пины версий + integrity — **не менять** скрипт-теги).
- Без сборки: `.jsx` файлы подключаются `<script type="text/babel">`, общие компоненты экспортируются через `Object.assign(window, {...})` (у Babel-скриптов раздельные скоупы!).
- Стили: один `app.css`, CSS-переменные, container queries, никакого Tailwind.
- Состояние: один корневой `App` + React Context (`AppCtx` / `useApp()`), персист в `localStorage["spotless_proto_v1"]`.

### 5.2. Файлы

### 5.3. Адаптив — выстраданные правила
Прототип обязан выдерживать **экстремальное сжатие** (тайлинг-WM, пол-экрана браузера):

1. Правая панель прячется **первой** (`@media max-width:1300px`) — раньше, чем начнёт деградировать плеер.
2. Сайдбар: кнопка ручного сворачивания в рейл (иконка-панелька у лого, состояние в LS) + авто-рейл `<1180px`. Класс `.rail` на `.shell`, ширина через `--sbw`.
3. Островок: контейнер-запросы (`container-type: inline-size`), центр `clamp(150px, 46cqi, 520px)`, ступени скрытия с `!important` (inline-стили иначе перебивают): громкость ≤1080 → лайк/＋/$ ≤980 → мик/устройства ≤790 → shuffle/repeat/очередь ≤545 → тайминги+название ≤400 → prev/next ≤330. На минимуме остаются обложка+play+разворот. Название трека `.isl-who { min-width:72px }` — приоритетнее второстепенных кнопок.
4. Строки треков: колонка альбома/прослушиваний и кнопка загрузки прячутся контейнер-запросами `main`.
5. Сетки карточек — `auto-fill/minmax`, шапки — `clamp()` + `flex-wrap`.

### 5.4. Грабли, на которые уже наступили (не повторять)
- **CSS-анимация с `fill-mode: both` перебивает inline `transform`** → попапы сдвигаются через `left: calc(50% + dx)` + `marginLeft`, НЕ через transform (конфликт с `.fade-up`).
- **Inline-стили перебивают media/container-правила** → ступени скрытия с `!important`; не ставить inline `display`/`min-width` на элементы, которыми управляет CSS.
- **Поповер с замером позиции** должен монтироваться только при открытии (отдельный компонент `PlusPop`), иначе `useLayoutEffect` отработает до открытия по `ref=null`.
- **Вложенные компоненты-обёртки в функции рендера** = перемонтирование и мигание каждый тик (онбординг) → все обёртки на верхнем уровне модуля.
- **Каждый `<script type="text/babel">` — свой скоуп** → экспорт через `window`, имена styles-объектов уникальные.
- Тикающий прогресс плеера ререндерит всё дерево — следить, чтобы тяжёлые поддеревья не зависели от `player.pct` без нужды.

### 5.5. Будущая реализация (вне прототипа)
Это контекст для текстов и UX-решений. **В прототипе ничего из этого не кодится** — но тексты и экраны не должны этому противоречить.

**Решения зафиксированы (июнь 2026), не пересматривать молча:**

- **Клиент** — нативное десктоп-приложение, **Rust + Slint** (пишется с Claude Code по этому прототипу). Slint выбран вместо iced/libcosmic осознанно: декларативная разметка переносит CSS-переменные тем в Slint-глобалы почти один в один, API стабилен с 1.0 — а у iced API капитально менялся между версиями (0.10→0.13), из-за чего LLM-генерация без человека-читателя смешивает несовместимые идиомы. GPL-вариант Slint совместим с AGPL проекта. **Telegram Mini Apps отменены** — никаких веб/webview-обёрток, основа — приложение.
- **Ядро клиента** (внутри процесса приложения): rust-libp2p (p2p-раздача), symphonia + cpal (аудио), некастодиальный кошелёк, верификация чанков, локальная база.
- **Внешняя инфраструктура** (работает, когда клиенты выключены) — **Go**, пишется владельцем проекта вручную: bootstrap/relay-ноды (NAT-пробитие), индексатор каталога для поиска, кабинет артиста. Контент на серверах **не хранится**. go-libp2p ↔ rust-libp2p интероперабельны штатно (фундамент IPFS) — Go-ноды и Rust-клиенты в одной сети это проверенный расклад.
- **Деньги** — блокчейн **Solana** (решение: TON держался на Telegram-интеграции и отпал вместе с Mini Apps; его асинхронная акторная модель контрактов — худший вариант для денежной логики). Контракты — **Rust / Anchor**. Внутренний актив — **USDC на Solana** (цены в UI долларовые → актив долларовый, волатильность исключена). Распределение проектируется как накопление в контракте + claim/меркл-батчи, **не** «N переводов каждому артисту от каждого пользователя». Комиссии — доли цента, финальность — доли секунды. Газ спонсируется через отдельного fee payer — пользователь слова «газ» не видит.
- **Фиат** — без собственного платёжного сервера: он-рамп-провайдеры (агрегатор Onramper либо связка MoonPay / Ramp / Mercuryo по географии), методы Apple Pay / Google Pay / карта / SEPA; PayPal — один из методов внутри виджета. KYC делает провайдер, один раз. Подписочного API у он-рампов нет → автосписание заменено авто-напоминанием (см. 1.2).
- **Лицензия**: AGPL.
- **Контент**: «названия треков оригинальные — будут, когда проект выйдет, после договора с правообладателями».

**Право — инварианты для формулировок:**

- Архитектура строго **некастодиальная**: проект никогда не держит ключи и средства пользователей и не встаёт посредником в переводах. Пока это так — проект вне периметра CASP по MiCA (ЕС) и вне money transmitter по FinCEN (США). Любая новая «централизованная» фича сначала проверяется на этот инвариант.
- KYC существует только на границах фиат↔крипта и выполняется лицензированными провайдерами: он-рамп на стороне слушателя, офф-рамп на стороне артиста при выводе в фиат. Внутри крипто-контура KYC нет — карточка «подтверждение личности не понадобится» юридически корректна.
- Перевод с биржи на свой кошелёк >€1000 проходит верификацию владения **на стороне биржи** (travel rule) — это трение чужого экрана, не нашего.
- Регуляторика подвижна (ревизия правил self-hosted-кошельков в ЕС идёт прямо сейчас) — перед запуском в ЕС/США этот блок перепроверяется с юристом. В текстах продукта не давать обещаний из области права («анонимно», «никогда не потребуют») — только фактические формулировки («не понадобится» про наш интерфейс).

## 7. Рабочий процесс с заказчиком

- Заказчик гоняет прототип руками (в т.ч. экстремальный тайлинг) и присылает списки правок со скриншотами — **каждая мелочь важна**, планка качества «выверять чуть ли не каждый пиксель».
- Правки вносить точечно, после каждой партии — прогон проверяющим по конкретному сценарию.
- Большие переделки — новыми версиями файлов, мелочи — в текущих.
- Не переходить к следующему крупному пункту, пока текущий не доведён («пока не закончим прям полностью — дальше не идём»).
